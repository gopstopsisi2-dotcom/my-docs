\# Стратегия версионирования документации



\## Принципы



Документация следует версионированию продукта. Каждой версии продукта соответствует версия документации с тем же номером.



\## Типы изменений



\### Мажорные изменения (major)



Примеры:

\- Переработка архитектуры

\- Удаление или изменение ключевых API

\- Новая структура документации



Обозначение в тегах: v2.0.0, v3.0.0



\### Минорные изменения (minor)



Примеры:

\- Добавление новых разделов

\- Дополнение существующих инструкций

\- Исправление неточностей



Обозначение в тегах: v1.1.0, v1.2.0



\### Патчи (patch)



Примеры:

\- Исправление опечаток

\- Обновление примеров кода

\- Незначительные правки



Обозначение в тегах: v1.0.1, v1.0.2



\## Поддержка версий



\- LTS-версии (Long Term Support): поддержка документации 24 месяца

\- Обычные версии: поддержка 6 месяцев



\## Процесс обновления документации



1\. Разработчик создает новый тег в Git

2\. CI/CD автоматически запускает сборку

3\. Mike публикует новую версию на gh-pages

4\. Пользователи переключаются между версиями через селектор на сайте



\## Команды для работы с mike



\- Просмотр всех версий: mike list

\- Публикация новой версии: mike deploy v2.0.0 -u

\- Удаление версии: mike delete v1.0.0

\- Установка версии по умолчанию: mike set-default v1.0.0



\## Связь с Git тегами



В репозитории созданы теги:

\- v1.0.0

\- v1.1.0

\- v2.0.0



Каждый тег соответствует версии документации, которая была актуальна на момент выпуска продукта.
